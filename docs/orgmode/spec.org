#+title: The con File Format Specification
#+date: 2026-03-25
#+options: toc:2 num:t

* Specification

- Version :: 2
- Date :: 2026-03-25
- Status :: Stable

This document defines version 2 of the con file format specification.
It supersedes all prior informal descriptions. New implementations SHOULD
target this version. Existing implementations SHOULD upgrade to support
version 2 features, particularly the =atom_id= field.

The keywords MUST, MUST NOT, REQUIRED, SHALL, SHOULD, SHOULD NOT, MAY,
and OPTIONAL follow [[https://www.rfc-editor.org/rfc/rfc2119][RFC 2119]] semantics.

* Overview

The con file format stores atomic configurations for molecular dynamics
and transition-state search simulations. It originated in the [[https://eondocs.org][eOn code]] and
is now used across multiple computational chemistry tools including
[[https://wiki.fysik.dtu.dk/ase/][ASE]] for structure manipulation.

A ~con~ file contains one or more /frames/. Each frame encodes a
simulation cell, per-type metadata (masses, atom counts), and
per-atom data (coordinates, constraints, identity). The .convel
extension adds per-atom velocity data after each frame's coordinates.

* File extensions

- =.con= :: Coordinate-only configuration files.
- .convel :: Configuration files with velocity data appended per frame.

* Encoding

~con~ files MUST use UTF-8 encoding. Line endings MAY be LF (=\n=) or
CRLF (=\r\n=). Parsers MUST accept both. All numeric values use
ASCII decimal representation (no locale-dependent formatting).

* Frame structure

Each frame consists of:

1. A 9-line header.
2. One coordinate block per atom type.
3. (.convel only) A blank separator line followed by one velocity block per atom
   type.

Multiple frames are concatenated directly with no inter-frame
separator. A parser detects a new frame by encountering a valid
9-line header after the previous frame's data.

* Header (9 lines)

Every frame begins with a fixed 9-line header.

| Line | Name              | Content                        | Example                |
|------+-------------------+--------------------------------+------------------------|
|    1 | Generator comment | Free-form text                 | =Generated by eOn=       |
|    2 | Metadata          | JSON object or free-form text  | ={"con_spec_version":2}= |
|    3 | Cell dimensions   | 3 floats: Lx Ly Lz             | =15.3456 21.702 100.0=   |
|    4 | Cell angles       | 3 floats: alpha beta gamma     | =90.0 90.0 90.0=         |
|    5 | Reserved          | Free-form text (round-tripped) | =0 0=                    |
|    6 | Reserved          | Free-form text (round-tripped) | =218 0 1=                |
|    7 | Atom type count   | 1 integer: N                   | =2=                      |
|    8 | Atoms per type    | N integers                     | =216 2=                  |
|    9 | Mass per type     | N floats (atomic mass units)   | =63.546 1.00793=         |

** Line 1: Generator comment

Free-form text identifying the tool that produced the file. Writers
SHOULD set this to a human-readable identifier (e.g., =Generated by
eOn=, =readcon-core=). Parsers MUST preserve this line through
read-write cycles but MUST NOT assign it semantic meaning.

** Line 2: Metadata

Line 2 carries machine-readable metadata as a single-line [[https://www.json.org][JSON]] object.

*** Version 2+ files

A version 2 (or later) writer MUST emit a JSON object on line 2
containing at least the key =con_spec_version= with an integer value:

#+begin_example
{"con_spec_version":2}
#+end_example

Additional keys MAY appear in the object for forward-compatible
extension:

#+begin_example
{"con_spec_version":2,"generator":"readcon-core 0.6.0","timestamp":"2026-03-25T12:00:00Z"}
#+end_example

Parsers MUST preserve unrecognized keys through round-trips.

*** Legacy (pre-v2) files

Files produced before this specification may contain free-form text on
line 2 (e.g., =Time=, =0.0000 TIME=, or an empty line). A conforming
parser detects the format by checking whether line 2, after stripping
leading and trailing whitespace, starts with the ={= character:

- Starts with ={=: parse as JSON, extract =con_spec_version=.
- Does not start with ={=: treat as a legacy file with an implicit
  =spec_version= of 1. No error.

If line 2 starts with ={= but contains malformed JSON, the parser
MUST report an error.

If the JSON object lacks =con_spec_version=, the parser MUST report
an error.

If =con_spec_version= exceeds the highest version the parser
supports, the parser MUST report an error rather than silently
misinterpreting the data.

** Lines 3-4: Cell geometry

Line 3 contains three whitespace-separated floats: the cell edge
lengths Lx, Ly, Lz in angstroms.

Line 4 contains three whitespace-separated floats: the cell angles
alpha, beta, gamma in degrees.

Tabs and spaces are both valid separators throughout the format.

** Lines 5-6: Reserved

These lines are free-form text with no defined semantics. Older eOn
versions stored simulation state here (iteration counters, time step
data). Writers MAY emit empty lines. Parsers MUST read and preserve
these lines for round-trip fidelity but MUST NOT depend on their
content.

** Lines 7-9: Type metadata

Line 7: a single positive integer N, the number of distinct atom
types in this frame.

Line 8: exactly N whitespace-separated positive integers, giving the
atom count for each type.

Line 9: exactly N whitespace-separated floats, giving the atomic mass
(in atomic mass units) for each type.

* Coordinate blocks

After the 9-line header, for each atom type /i/ (1 to N) in the order
declared in lines 8-9:

1. *Symbol line*: the chemical symbol for this type (e.g., =Cu=,
   =H=). Trailing whitespace is ignored.
2. *Label line*: =Coordinates of Component /i/=. Parsers SHOULD accept
   minor variations in capitalization and spacing (e.g., =component=
   vs. =Component=, variable spacing around the number).
3. *Atom lines*: one line per atom in this type, containing:

| Column | Type | Description |
|--------+-------+--------------------------------------|
|      1 | float | x coordinate (angstroms) |
|      2 | float | y coordinate (angstroms) |
|      3 | float | z coordinate (angstroms) |
|      4 | int | Fixed flag: 0 = free, 1 = frozen |
|      5 | int | Atom index (see [[#atom-index]]) |

Columns are whitespace-separated. Tabs and spaces are both valid.

** Atom index (column 5)
:PROPERTIES:
:CUSTOM_ID: atom-index
:END:

The atom index preserves the original position of each atom in the
input structure before type-based grouping. This field solves a
fundamental problem: the ~con~ format groups atoms by element type,
which reorders them relative to their original sequence. Without a
persistent index, the original ordering cannot be recovered after a
read-write cycle.

*** Spec version 2 requirements

- Writers MUST emit column 5 on every atom line.
- Column 5 MUST contain the original atom index (the position of this
  atom before type-based reordering). Writers MUST NOT substitute a
  sequential counter that does not reflect the true pre-grouping
  order.
- Readers MUST parse and store column 5.
- Readers MUST preserve the stored value through write-back.

*** Spec version 1 behavior

- Column 5 is present and contains a non-negative integer, but its
  value carries no defined semantics. A sequential counter is typical.
- Readers MAY ignore column 5.

*** Backward compatibility for column 5

Readers SHOULD accept 4-column atom lines for interoperability with
hand-edited or third-party files. When column 5 is absent, the reader
MUST default to the sequential position within the frame (0, 1, 2,
...).

A version 2 reader processing a version 1 file (sequential counters)
produces correct results: sequential values constitute a valid
atom_id for structures that were never reordered.

*** Why atom_id matters

A structure with atoms in order C, C, C, O, C, C (indices 0-5) gets
written as five C atoms followed by one O atom. Without a meaningful
column 5, the original ordering vanishes. The atom_id field
preserves the pre-grouping index so that the original atom sequence
can be reconstructed after any number of read/write cycles.

This property matters for comparing geometries against reference
structures (e.g., [[https://github.com/HaoZeke/nebmmf_repro][Baker transition-state test set]]) after [[https://arxiv.org/abs/2601.12630][NEB]] or [[https://doi.org/10.1002/cphc.70329][dimer]]
calculations that read and rewrite ~con~ files.

* Velocity blocks (convel only)

In =.convel (coordinates + velocities)= files, a velocity section follows the coordinate blocks
of each frame:

1. *Blank separator*: one empty line (or whitespace-only) marking
   the boundary between coordinates and velocities.
2. For each atom type /i/ (1 to N):
   a. *Symbol line*: same chemical symbol as in the coordinate block.
   b. *Label line*: =Velocities of Component /i/=
   c. *Atom lines*: one line per atom: =vx vy vz fixed_flag atom_id=

The velocity fields =vx=, =vy=, =vz= are floating-point velocity
components. The =fixed_flag= and =atom_id= follow the same rules
as in coordinate blocks.

If no blank line follows the coordinate blocks, the frame has no
velocity data (standard =.con= behavior). A parser SHOULD detect this
by peeking at the next line: if it is blank (or only whitespace), a
velocity section follows; otherwise the current frame ends and a new
frame header (or end-of-file) follows.

* Multi-frame files

Multiple frames are concatenated with no explicit separator. After
parsing a frame's coordinate blocks (and velocity blocks, if present),
the parser attempts to read the next 9-line header. If fewer than 9
lines remain, parsing ends.

Writers MUST NOT insert extra blank lines or separators between
frames. The only blank line in the format is the velocity separator
within a =.convel (coordinates + velocities)= frame.

* Data types and precision

- *Floats*: any valid decimal floating-point representation. Writers
  SHOULD emit at least 6 significant decimal digits for coordinates
  and masses. For lossless f64 round-tripping, 17 decimal digits
  suffice.
- *Integers*: =fixed_flag= is 0 or 1. =atom_id=, =natm_types=,
  and =natms_per_type= values are non-negative integers representable
  as unsigned 64-bit values.

* Constraints and limits

- Atoms MUST appear grouped by type, in the order declared in header
  lines 8-9.
- Component numbering in label lines starts at 1.
- The total atom count equals the sum of values on line 8.
- Symbol strings SHOULD match standard chemical element symbols ([[https://iupac.org][IUPAC]]
  notation, 1-2 characters, first character uppercase).
- No upper limit on atom types or atom count is imposed by this
  specification. The eOn implementation historically supported up to
  100 types; other implementations may differ.

* Querying the spec version

** From the file

Read line 2. If it starts with ={=, parse as JSON and extract
=con_spec_version=. Otherwise the file predates this specification
(implicit version 1).

** From the library

| Language    | Compile-time                   | runtime                           |
|-------------+--------------------------------+-----------------------------------|
| Rust        | =readcon_core::con_SPEC_VERSION= | same                              |
| C/C++       | =#define RKR_con_SPEC_VERSION=   | =rkr_con_spec_version()=            |
| C/C++       | N/A                            | =rkr_frame_spec_version(h)=         |
| C/C++       | N/A                            | =rkr_frame_metadata_json(h)=        |
| Python      | N/A                            | =readcon.con_SPEC_VERSION=          |
| Python      | N/A                            | =frame.spec_version=                |
| Python      | N/A                            | =frame.metadata=                    |
| Cap'n Proto | N/A                            | =specVersion= field on =ConFrameData= |

C/C++ compile-time gating example:

#+begin_src c
#include <readcon-core.h>
#if RKR_con_SPEC_VERSION < 2
#error "readcon-core spec v2 required for atom_id support"
#endif
#+end_src

* Recommended metadata keys

The JSON metadata line accepts arbitrary keys. This section documents
keys that implementations SHOULD recognize when present. All keys are
OPTIONAL; omitting a key carries no penalty. Unrecognized keys MUST
be preserved through round-trips.

| Key           | Type   | Description                                                   |
|---------------+--------+---------------------------------------------------------------|
| =generator=   | string | Tool name and version that wrote this file.                   |
| =units=       | object | Unit system (see below).                                      |
| =frame_index= | int    | Zero-based frame number within a trajectory.                  |
| =time=        | float  | Simulation time of this frame, in the declared time unit.     |
| =timestep=    | float  | Integration timestep, in the declared time unit.              |
| =neb_bead=    | int    | Bead (image) index within an NEB band.                        |
| =neb_band=    | int    | Band index within an optimization path.                       |

** Units

The =units= key holds an object mapping physical dimensions to unit
strings. Omitted dimensions default to the eOn conventions shown
below.

| Dimension  | Default   | Examples                                  |
|------------+-----------+-------------------------------------------|
| =length=   | angstrom  | =angstrom=, =bohr=, =nm=                 |
| =mass=     | amu       | =amu=, =kg=, =electron_mass=             |
| =time=     | fs        | =fs=, =ps=, =au=                         |
| =energy=   | eV        | =eV=, =hartree=, =kcal/mol=, =kJ/mol=   |
| =velocity= | (derived) | Defaults to =length / time=. Override for non-SI-derived units. |

Example metadata line with units and trajectory info:

#+begin_example
{"con_spec_version":2,"generator":"eOn 3.1","units":{"length":"angstrom","energy":"eV"},"frame_index":42,"time":21.0,"timestep":0.5}
#+end_example

Implementations that do not perform unit conversion SHOULD still
preserve the =units= key so that downstream tools can interpret the
data correctly.

** Frame ordering in trajectories

Multi-frame files produced by =cat *.con > traj.con= lose filesystem
ordering. Writers that produce trajectories SHOULD include
=frame_index= (and =time= when available) so that consumers can
reconstruct the correct sequence without relying on file-system
metadata.

For NEB calculations, =neb_bead= and =neb_band= identify the
position of each image within the path optimization. These keys
complement =frame_index= (which tracks the global position in the
file) rather than replacing it.
